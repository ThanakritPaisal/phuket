Teaching LOMA who to recommend, to whom.

Start here

You log what tourists do. You don't yet log what they saw.

Every event we record — a thumbs-up, a tapped route, a confirmed visit — is an action someone took. To rank places, a model also needs the opposite: the cards a tourist was shown and ignored. Without that, there are no negatives and no way to correct for the fact that the top card is seen far more than the sixth.

The one unlock

Add an rec_impression event — fired when a card is shown, carrying its position and the tourist's context at that moment.

That single addition converts the logs from “a pile of clicks” into trainable rows of (context, item, shown → outcome). It mirrors the thumbs implementation and is the prerequisite for everything below.

Labels

Turning mixed signals into one target

A thumbs-up and an actual visit are not worth the same. We fold every logged signal into a graded relevance score, then down-weight by card position (a top-slot no-click is a stronger negative than a bottom-slot one) and by recency.

Where personalization actually lives — the cross features. nationality × category affinity, age × price, needs ↔ item match, hotel × category. Tree models learn these splits directly; neural models learn them as embeddings. Your hand-tuned loma_score isn't discarded — it enters as a feature and the model learns the residual on top.

The algorithms

A maturity ladder, matched to how much data exists

Don't reach for a neural net on day one. Each rung ships value on its own and earns the next by accumulating the data that rung needs.

0
Phase 0 · today · ~no data
Bayesian-smoothed demographic priors

Segment affinity table

Per nationality × category (and age × price) up-vote rates with shrinkage toward the global mean so three votes don't crown a category. Solves cold-start and ships now — it's just aggregation over the thumbs you already collect.

reads  rec_feedback + demographics returns  a reweight over loma_score
1
Phase 1 · hundreds–thousands of labeled impressions
Gradient-boosted trees — LightGBM / LambdaMART

The workhorse ranker

Pointwise (logistic) to start, then pairwise LambdaMART to optimize rank order directly. The sweet spot for sparse tabular data with rich cross features and modest volume — fast to train, hard to beat here, and it explains itself. Retrain nightly.

reads  full feature vector + graded labels returns  calibrated P(match) per provider
2
Phase 2 · 10k+ interactions
Two-tower neural embeddings + GBDT re-rank

Retrieval at scale

A tourist tower and a provider tower learn embeddings so hundreds of operators can be retrieved by vector similarity in milliseconds, with the Phase-1 tree re-ranking the shortlist. Only worth it once catalog and traffic grow.

reads  interaction history + content features returns  candidate shortlist + embeddings
3
Phase 3 · steady traffic
Contextual bandit — Thompson sampling / LinUCB

Closing the loop

The system chooses what to show, so it must explore, not just exploit — otherwise today's popular places get shown more, get more votes, and win forever. The bandit balances the two and logs each pick's propensity so we can evaluate new models offline. This matters more than model sophistication.

reads  ranker scores + uncertainty returns  the shown set + propensities

Guardrails

Pitfalls we design against — most of them LOMA-specific

A naive engagement optimizer would quietly betray the product's mission. These are constraints, not afterthoughts.

Runaway loop

Rich-get-richer

Popular places get shown → get more votes → get shown more, burying the hidden gems that are LOMA's whole reason to exist.

Fix · bandit exploration + an explicit hidden-gem exposure floor.
Mission risk

Operator fairness

“100% to the owner” means many small operators. Pure engagement collapses onto a handful of winners.

Fix · exposure-fairness / diversity term so the long tail keeps impressions.
Bias

Position bias

The top card is clicked more regardless of quality; ignore it and the model just learns “position 1 = good.”

Fix · log position, correct with inverse-propensity weighting.
Sparsity

One-shot tourists & new operators

Most visitors never return; new operators start with zero interactions.

Fix · personalize on segments, not user IDs; lean on content features for cold items.
Measuring it. Offline — NDCG@3 (you show ~3 picks), AUC, and replay/off-policy estimates using logged propensities before anything goes live. Online — A/B on thumbs-up rate and route-clicks, with the north star being confirmed-visit rate, plus the fairness and hidden-gem-exposure guardrails as release gates. Privacy stays intact: we train on coarse bands (age ranges, nationality), never PII.

Near-term path

Four moves, in order

Add rec_impression logging

Position + context snapshot on every card shown. The unblocker — small patcher, mirrors the thumbs work.

Harden the best labels collecting now

Keep thumbs; make sure provider_confirmed_visit and card-dwell fire reliably.

Ship Phase-0 priors for an immediate lift

A GET /recommendations that reweights by the nationality × category affinity table — usable after weeks, not months.

Train the LightGBM ranker, evaluate by replay, then A/B behind the bandit

With the hidden-gem and fairness guardrails wired in as release gates.
