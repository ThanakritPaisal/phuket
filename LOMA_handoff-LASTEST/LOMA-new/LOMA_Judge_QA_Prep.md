## 🔴 TIER 1 — ถามแน่ 100% และตอบพลาดคือจบ

### 1. "AI อยู่ตรงไหน? นี่มันแค่ if-else ให้คะแนน"
**ทำไมอันตราย:** ชื่องานคือ *AI for Better Tourism*. Scoring engine ของเราคือ deterministic weights (0.30·L + 0.25·Q + …) ไม่มี model ไม่มี LLM. Chatbot ก็ keyword rules.

**ตอบ:** "ตอนนี้เป็น *rule-based scoring* ครับ และผมเลือกแบบนี้ตั้งใจ — เพราะ 3 เหตุผล: (1) มัน **explainable** ทุกคะแนนบอกได้ว่ามาจากสัญญาณไหน ซึ่งจำเป็นมากเวลาเราจะถอดร้านออกจากระบบ (2) มัน **audit ได้** ไม่ใช่ black box ที่ชุมชนโต้แย้งไม่ได้ (3) ที่ 83 ร้าน model ยังไม่มีข้อมูลพอจะเทรน
ส่วน AI จริงเข้า 2 จุดที่ชัดเจน: **NLP อ่านเนื้อรีวิว** สกัดธีม (สะอาด/บริการ/คุ้ม/โกง) — อันนี้ rule ทำไม่ได้ ต้องใช้โมเดล; และ **recommendation engine** ที่เทรนจาก เพศ/อายุ/สัญชาติ + เวลาที่เปิดดู + หมวดที่ดู
ผมไม่อยากเรียกมันว่า AI ทั้งที่มันยังไม่ใช่ครับ — วันนี้คือ **AI-assisted curation ที่ human confirm**"

**⚠️ ห้ามพูด:** "ระบบ AI ของเราวิเคราะห์…" เพราะพอโดนเจาะแล้วพัง

### 2. "ไม่เก็บคอมมิชชั่น แล้วบริษัทอยู่รอดยังไง? ใครจ่ายเงินให้คุณ"
**ทำไมอันตราย:** เราชูจุดขาย "No hidden commission" ทั้งเว็บ — แต่ไม่มี revenue model ที่ไหนเลย นี่คือ**คำถามฆ่า**

**ตอบ (ต้องเลือกให้ได้ก่อนขึ้นเวที):**
- **SaaS กับโรงแรม** — เก็บรายเดือน/ต่อห้อง โรงแรมจ่ายเพราะ LOMA ลดงาน front desk + ได้รายงาน ESG/local impact ไปใช้กับ OTA และเชนแม่
- **ขายข้อมูล/รีพอร์ตให้ภาครัฐ** — ททท./อบจ./สสว. อยากรู้ว่าเงินท่องเที่ยวไหลลงท้องถิ่นแค่ไหน เรามีข้อมูลนั้นเป็นเจ้าเดียว
- **งบ CSR/CSV จากองค์กร** — เชนโรงแรมใหญ่มีงบ sustainability อยู่แล้ว
- **ทุน/กองทุนพัฒนาชุมชน** ช่วงแรก

**ประโยคปิด:** "เราไม่เก็บคอมฯ จาก *ผู้ประกอบการ* เพราะนั่นคือสิ่งที่ทำให้คำแนะนำเชื่อถือได้ แต่เราเก็บเงินจาก *ฝั่งดีมานด์* ที่ได้ประโยชน์ — โรงแรม และภาครัฐ"

### 3. "Impact Credits เอาไปแลกอะไรได้? ถ้าแลกไม่ได้ โรงแรมจะสนใจทำไม"
**ทำไมอันตราย:** ตอนนี้มันเป็นแค่ **แต้มกับป้าย tier** — ไม่มีมูลค่าจริง โรงแรมได้ Gold แล้วไง

**ตอบ:** "Credits ไม่ใช่เงินครับ มันคือ **หลักฐาน** — แล้วหลักฐานนี้แปลงเป็นของจริง 3 ทาง:
(1) **ป้าย/ใบรับรอง 'Phuket Local Impact Leader'** ติดหน้าโรงแรม + ใส่ในโปรไฟล์ Booking.com/Agoda → นักท่องเที่ยวสาย responsible travel เลือกโรงแรมจากตรงนี้
(2) **รายงาน ESG** ที่เชนโรงแรมและนักลงทุนต้องส่งอยู่แล้ว — วันนี้เขาไม่มีตัวเลข local sourcing ให้ใส่เลย
(3) ระยะยาว: สิทธิ์เข้าถึงชุมชน/ประสบการณ์ที่ tier สูงเท่านั้นที่แนะนำได้
เราจงใจไม่ให้เป็นเงินสด เพราะพอเป็นเงินเมื่อไหร่ มันก็กลายเป็นค่าคอมฯ แบบเดิม — และความน่าเชื่อถือตายทันที"

### 4. "ทำไมโรงแรมจะทิ้ง tour desk ที่ได้ค่าคอมฯ อยู่แล้ว มาใช้ของคุณ"
**ทำไมอันตราย:** เราไปแย่งรายได้เขา — และเราให้แต้มที่กินไม่ได้แทน

**ตอบ:** "เราไม่ได้ขอให้เขาทิ้งครับ — **LOMA ไม่แข่งกับ tour desk** tour desk ขายทัวร์เกาะ ทัวร์เรือ ของราคาหลักพัน ส่วน LOMA ตอบคำถาม *'กินข้าวเย็นที่ไหนดี'* ซึ่งเป็นคำถามที่ tour desk ไม่ได้เงินอยู่แล้ว และเป็นภาระ front desk เปล่าๆ
เราเข้าไปในช่องว่างที่**ไม่มีใครได้เงิน** และเปลี่ยนมันเป็นภาพลักษณ์ + ลดงาน — ไม่ได้ไปแตะรายได้เดิมเลย"

### 5. "ข้อมูล 83 ร้านนี่มาจากไหน? scrape จริงไหม? Google ToS ห้าม scrape นะ"
**ทำไมอันตราย:** ข้อมูลเราเป็น mock/seed ทั้งหมด และเราอ้าง Google Maps/TripAdvisor/DBD บนสไลด์

**ตอบ:** "ใน demo วันนี้เป็น **seed data** ครับ ผมไม่บอกว่ามันคือข้อมูลจริงที่ verify แล้ว
ของจริงเราจะใช้ **Google Places API** ซึ่งเป็น official API ไม่ใช่ scraping — ดึง rating/review count/hours/พิกัดได้ถูกกฎหมาย, **DBD open data** สำหรับทะเบียนธุรกิจ, และ **ข้อมูล CBT ของ อพท./ททท.** ที่เปิดสาธารณะอยู่แล้ว (ในระบบมี 22 ชุมชนจากชุดนี้จริง)
ส่วนเนื้อรีวิวที่ต้องอ่านด้วย NLP — อันนั้นต้องคุยเรื่องสิทธิ์ ยอมรับว่ายังไม่จบ"

## 🟠 TIER 2 — โดนแน่ ตอบได้ถ้าเตรียม

### 6. "รู้ได้ยังไงว่านักท่องเที่ยวไปร้านจริง"
**ช่องโหว่จริง:** `provider_confirmed_visit` มาจาก**นักท่องเที่ยวกดปุ่มเอง** — ปลอมได้

**ตอบ:** "วันนี้เป็น self-report ครับ ซึ่งอ่อน ผมยอมรับ ของจริงมี 3 ชั้น: **ร้านสแกน referral code** ของแขก (ฟีเจอร์นี้มีในระบบแล้ว), ร้านกดยืนยันจาก provider portal, และ cross-check กับ direction click + เวลา
แต่จุดสำคัญคือ — **confirmed visit ให้ 20 แต้ม ส่วน scan ให้ 2** เราตั้งใจให้ของที่ปลอมง่ายมีค่าน้อย"

### 7. "พนักงานโรงแรมปั๊มแต้มเองได้ไหม"
**มีในระบบแล้ว:** session dedupe — event เดียวกัน + provider เดียวกัน + session เดียวกัน นับครั้งเดียว, ปั๊ม QR ถูก cap, และหน้า admin **แสดงบนจอ**ว่าจับได้กี่ครั้งและไม่ให้แต้ม

**ตอบเพิ่ม:** "และเพราะแต้มแลกเงินไม่ได้ (ข้อ 3) แรงจูงใจในการโกงมันต่ำมากอยู่แล้ว — โกงมาแล้วได้ป้ายปลอมที่ถูกออดิทได้"

### 8. "เกณฑ์ Hidden Gem (locality≥78, quality≥72, gap≥55…) ใครตั้ง? validate ยังไง"
**ช่องโหว่จริง:** ตัวเลขพวกนี้**ผมตั้งเอง** ไม่มี ground truth

**ตอบ:** "ตั้งจากการ calibrate กับ catalogue ปัจจุบันครับ ยังไม่มี ground truth — และนี่คือสิ่งที่ pilot ต้องทำ: ให้ human curator ตัดสิน 200 ร้าน แล้ววัด precision/recall ของเกณฑ์ ปรับน้ำหนักจากตรงนั้น
สิ่งที่ผม *ยืนยันได้* คือ **ตรรกะ** ไม่ใช่ตัวเลข: hidden gem ต้องไม่ใช่การให้รางวัลกับความดัง"

### 9. "ภูเก็ตมี 42,000 ร้าน คุณมี 83 — scale ยังไง ใครคัด"
**ตอบ:** "AI คัดจากหลายร้อยเหลือ shortlist หลักสิบ คนเช็คเฉพาะหัวแถว — ที่ 1,000 ร้าน คนหนึ่งคนไหว ที่ 42,000 ไม่ไหว
แต่เราไม่ต้องการ 42,000 ครับ **เราต้องการ 300 ร้านที่ดีที่สุด** โรงแรมไม่ได้ต้องการไดเรกทอรี เขาต้องการของที่กล้าแนะนำ — คุณค่าของเราคือ**ตัวกรอง ไม่ใช่ความครอบคลุม**"

### 10. "ต่างจาก Google Maps / TripAdvisor / Klook ยังไง เขาทุนหนากว่าเยอะ"
**ตอบ:** "Google Maps ใครก็ขึ้นได้ ไม่มีคัดกรอง · TripAdvisor อันดับบนมักซื้อพื้นที่ + เอนเอียงหาเจ้าดัง · Klook ขายทัวร์ที่จ่ายคอมฯ ได้ ซึ่งชุมชนเล็กจ่ายไม่ไหว
LOMA ต่างที่ **จุดส่งมอบ** — เราไม่ได้แข่งให้คนมาเสิร์ช เราส่งของถึงมือตอนที่พนักงานโรงแรมกำลังตอบคำถามพอดี **นั่นคือช่องที่เจ้าใหญ่เข้าไม่ถึง เพราะเขาไม่มีความสัมพันธ์กับ front desk**"

### 11. "เก็บ เพศ/อายุ/สัญชาติ ของนักท่องเที่ยว — PDPA ผ่านไหม"
**ช่องโหว่จริง:** ยังไม่มี consent flow ในเว็บเลย

**ตอบ:** "ยอมรับว่ายังไม่ได้ทำ consent screen ครับ ของจริงต้องมี: opt-in ก่อนเก็บ, เก็บเป็น **aggregate ไม่ผูกตัวบุคคล** (เราไม่ต้องรู้ว่าคุณคือใคร แค่รู้ว่า 'ผู้หญิงยุโรป 30-40 ตอนบ่าย มองหาคาเฟ่'), และไม่มี PII เลย — ไม่มีชื่อ ไม่มีอีเมล ไม่ต้อง login ซึ่งจริงๆ ทำให้ PDPA เบากว่าแอปทั่วไปมาก"

### 12. "ชุมชนพูดอังกฤษไม่ได้ นักท่องเที่ยวโทรไปแล้วคุยกันยังไง"
**ช่องโหว่จริง:** ฟอร์มมีช่อง "ภาษา" แต่เราไม่มีระบบแปล — inquiry ส่งไปแล้วจบตรงนั้น

**ตอบ:** "นี่คือช่องโหว่ที่ผมยอมรับ และเป็น**เหตุผลที่ AI แปลภาษาคือฟีเจอร์ถัดไป** — inquiry ควรถูกแปลอัตโนมัติสองทาง ตอนนี้ทางแก้ชั่วคราวคือ ฟอร์ม readiness มีช่อง 'ภาษาที่รองรับ' และเราแสดง badge ให้นักท่องเที่ยวเห็นก่อนติดต่อ + ชุมชนที่ไม่มีภาษาจะได้สถานะ *Contactable* ไม่ใช่ *Ready to Recommend*"

## ✅ 5 คำถามที่เราตอบได้ดีมาก — ลากเขามาตรงนี้ให้ได้

1. **"คัด local จริงยังไง"** → เกณฑ์ audit ได้ + gate ตัดแฟรนไชส์ + คน confirm (มีหน้าจอโชว์)
2. **"ร้านใหม่โผล่ตลอด จะตามยังไง"** → Freshness loop + ปุ่ม AI Refresh กดโชว์สดได้
3. **"AI จะเลือกแต่ร้านดังไหม"** → **ไม่** เพราะ Visibility Gap ทำให้ความดัง**หักคะแนน** — นี่คือหมัดเด็ด
4. **"ถ้านักท่องเที่ยวใช้เองได้ พนักงานจะยังใช้ทำไม"** → Passive vs Assisted (มีคำตอบอยู่บน UI เลย)
5. **"โรงแรมได้อะไร"** → Impact dashboard + tier + leaderboard กดโชว์สดได้

## 🎯 กฎ 3 ข้อตอนตอบ
